# 향후 고도화 항목

지금은 단순하게 구현했지만, 나중에 더 똑똑하게/편하게 만들 수 있는 항목을 기록한다.
(우선순위 낮음 — 핵심 기능 안정화 후 진행)

---

## 1. 가맹점 자동분류 고도화 (LLM / ML)

### 현재 구현 (2026-06, C-2)
- **방식**: 사용자 정의 규칙(`merchant_rules`)의 키워드가 가맹점 원문에 **부분 문자열로 포함**되면 해당 카테고리·별칭을 적용.
- **흐름**: 규칙은 비어서 시작 → 사용자가 1회 분류 → "규칙 저장" → 다음부터 자동 (learn-as-you-go).
- **위치**: `backend/app/main.py`의 `_classify()`, 가져오기 미리보기에서 적용.

### 한계
- **내장 지식 없음**: "스타벅스=카페"를 시스템이 원래 알지 못함. 사용자가 가르친 규칙만 안다.
- **오타·표기 변형 못 잡음**: `스타박스`, `STARBUCKS`는 `스타벅스` 키워드로 매칭 안 됨 (글자 그대로 비교).
- **의미 이해 없음**: `메가커피`는 '카페' 글자가 없어 '카페' 키워드 규칙으로 못 잡음.

### 고도화 방향 (쉬운 것 → 어려운 것)
1. **흔한 가맹점 규칙 기본 시딩** (가장 쉬움)
   - 명세서 4.5 예시표 + 한국 인기 가맹점(스타벅스/GS25/CU/배민/쿠팡/올리브영/카카오T 등)을 기본 규칙으로 제공.
   - 카테고리·결제수단 시딩처럼 `seed.py`에 추가하면 됨.
2. **일반 단어 키워드 활용** (구현 불필요, 사용 팁)
   - `카페`·`병원`·`편의점` 같은 공통어를 키워드로 등록 → 해당 단어 포함 가맹점 일괄 분류.
3. **ML 분류기**
   - 특성: 가맹점명·금액·요일·시간 등 → 카테고리 예측.
   - 학습 데이터: 사용자가 이미 분류한 거래(라벨)를 그대로 활용 (개인화).
4. **LLM 분류**
   - 가맹점명을 LLM에 보내 "이 가게는 어떤 카테고리?"를 분류 → 퍼지/의미 이해 가능 (메가커피 → 카페 등).
   - 로컬 LLM(Ollama) 활용 시 프라이버시·비용 유리.
5. **외부 가맹점 DB / 사업자번호 매칭**
   - 표준 업종분류와 연결.

### 적용 시 고려사항
- **프라이버시**: 외부(LLM/API)로 보낼 땐 최소 정보만(가맹점명). 계좌번호·잔액·금액은 제외. 로컬 LLM이면 더 안전.
- **폴백**: 자동 분류 실패 시 기존 키워드 규칙 → 수동 분류 순으로 폴백.
- **사용자 확인**: 자동 결과는 "추천"으로 표시하고 사용자가 확정 (오분류 방지).
- **규칙과 병행**: ML/LLM이 추천한 분류를 사용자가 확정하면 규칙으로도 저장 → 다음부턴 규칙이 먼저 처리(빠르고 무료).

## 2. 투자/자산 손익 추적

### 현재 (가계부 = 현금흐름)
- 가계부는 **통장에 들어오고 나간 거래**만 기록한다.
- 투자 입금 = 비소비(투자)로 통장에서 나감 → 현금흐름에 잡힘.
- 매도해서 돈이 돌아오면 = 수입 → **실현 손익**이 자연히 반영됨.

### 한계
- **평가 손익**(안 팔았는데 오르내린 것)은 "거래"가 아니라 "시세"라 가계부가 못 잡는다.
- 즉 "지금 내 주식이 얼마인지"는 현재 가계부 범위 밖.

### 고도화 방향
- **자산/포트폴리오 추적** 별도 도입: 보유 종목·수량·평가액(시세)을 주기적으로 기록 또는 증권사/시세 API 연동.
- 평가 손익·수익률·자산 배분(안전/위험) 시각화.
- 가계부(현금흐름)와 자산추적(평가액)은 데이터 성격이 달라 별도 모듈로.
- 로컬 자동화(시세 크롤링/LLM)와 엮을 수 있음.
