## 2026-06-24 — 1단계: Docker + 백엔드 기반 구축

### 한 것
- Docker Compose로 PostgreSQL(`db`) + FastAPI(`backend`) 2개 컨테이너 구성
- 명세서 6장 9개 테이블 정의(SQLAlchemy), 거래 CRUD API 구현
- DB GUI(TablePlus) 연결, Swagger(`/docs`)로 API 동작 확인

### 배운 것 / 삽질
- **도커 4대 개념**: 이미지(틀) / 컨테이너(붕어빵, 일회용) / 볼륨(데이터 금고, 영속) / 네트워크(컨테이너 간 통신).
- **함정 ① 한글 폴더명**: `docker compose up` 시 `project name must not be empty` 에러. 폴더명이 한글이라 자동 프로젝트명이 빈 값이 됨. → compose에 `name: ledger` 명시로 해결.
- **함정 ② Dockerfile 인라인 주석**: `FROM python:3.12-slim # 주석` → `FROM requires either one or three arguments`. Dockerfile은 명령어와 같은 줄에 `#` 주석 불가. → 주석을 별도 줄로 분리.
- **볼륨 영속성 직접 증명**: 테이블 생성 → `docker compose down`으로 컨테이너 삭제 → 다시 `up` → 데이터 생존 확인. `down -v`는 볼륨까지 삭제하므로 주의.
- **코드 vs 데이터 분리**: 컨테이너는 Dockerfile/compose로 재조립, 데이터는 볼륨에만 존재. → GitHub엔 레시피만 올라가고 실제 데이터는 로컬에만 남음. 백업은 `pg_dump`.

## 2026-06-25 — 거래내역 풀스택 슬라이스 + 깃 워크플로우

### 한 것
- 깃 규칙 확립: GitHub Flow, 커밋 컨벤션, PR 템플릿(`.github/`), GitHub 원격 연결 (PR #1)
- 기본 카테고리(41)·결제수단(5) 시딩, `GET /categories`·`/payment-methods`·`/tags`
- React+TS(Vite) 프론트엔드 셋업 + 컨테이너화 → 3계층(프론트–API–DB) 관통 (PR #2)
- 거래 입력 폼(날짜/유형/금액/카테고리/결제수단/가맹점/메모/태그) + 목록 화면

### 배운 것 / 삽질
- **수직 슬라이스 방식**: 한 기능(거래내역)을 DB→API→화면까지 끝까지 완성 후 다음으로. 혼자 프로젝트에 적합.
- **함정 ③ node_modules 마운트**: `./frontend:/app` 볼륨이 컨테이너 안 `node_modules`를 덮어씀. → 익명 볼륨 `/app/node_modules`로 가림 방지.
- **CORS**: 프론트(5173)↔백엔드(8000) 출처가 달라 브라우저가 차단. → FastAPI `CORSMiddleware`로 5173 허용.
- **태그 get-or-create**: 거래 저장 시 태그 이름으로 조회 후 없으면 생성·연결. 다대다(`transaction_tags`) 실전. 추후 태그별 집계 분석의 기반.
- **입력 경로 차이**: 수동 폼은 가맹점/태그를 직접 입력. CSV는 같은 `transactions` 표에 쌓이되 원본 가맹점명 보존·자동 분류가 핵심 (다른 경로, 같은 목적지).
