# 가계부 웹 앱 기능 명세서 (풀스택 / 확장형)

## 1. 개요

개인용 가계부 웹 앱. 학습을 겸한 프로젝트로, 제대로 된 3계층(프론트엔드–API–DB) 구조로 구현한다. 수입·지출을 빠르게 기록하고, 예산을 관리하며, 지출 패턴을 분석한다. 처음부터 데이터 분류·입력 경로의 확장을 전제로 설계해 후속 리팩토링을 최소화한다.

| 항목 | 내용 |
|------|------|
| 형태 | 풀스택 웹 앱 (프론트엔드 + REST API + DB) |
| 사용 대상 | 단일 유저로 시작, 멀티 유저 확장 대비 |
| 통화 | 원화(KRW) 기준 |
| 설계 원칙 | 분류·입력 경로 확장성 우선, 하드코딩 배제 |

---

## 2. 기술 스택

| 계층 | 기술 | 선택 이유 |
|------|------|----------|
| 프론트엔드 | React + TypeScript | 컴포넌트 기반 UI, 타입 안정성 |
| 백엔드 | Python + FastAPI | 자동 API 문서(Swagger), Pydantic 검증, 학습 가치 |
| DB | PostgreSQL | 관계형 구조에 적합, 거래/카테고리/예산 모델링 |
| ORM / 마이그레이션 | SQLAlchemy + Alembic | 스키마 버전 관리, 실무 패턴 |
| 검증 | Pydantic | 요청/응답 스키마 일원화 |

## 3. 핵심 기능

### 3.1 수입 / 지출 기록
- 거래 입력: 날짜, 금액, 유형(수입/지출), 카테고리, 결제수단, 메모
- 거래 목록: 최신순, 날짜별 그룹화
- 거래 수정 / 삭제
- 잔액 및 월 합계 자동 계산

### 3.2 카테고리 분류 (확장형)
- 기본 카테고리 제공 + 사용자 정의 추가/편집
- **계층 구조**: 대분류–소분류 (예: 식비 > 외식 / 식료품)
- 카테고리별 색상
- **자유 태그(tag)**: 거래에 다대다로 연결, 카테고리와 독립된 횡단 분류 축
  - 예: `데이트` 태그 — 외식·영화·선물 등 여러 카테고리에 흩어진 지출을 하나로 집계
  - 예: `정기권`, `여행`, `비상금` 등 새 축이 생겨도 태그만 추가 (구조 변경 불필요)
  - 카테고리 분석(외식 30만)과 태그 분석(데이트 23만)을 독립적으로 집계

### 3.3 예산 관리 (시각화 중심)
- 월 예산: 전체 / 카테고리별 (목표 금액, 설정은 선택)
- 예산 기준을 추상화해 카테고리뿐 아니라 태그·기간 등 다른 축으로도 확장 가능하게 설계
- **실시간 추적·강제 알림 없음**. 입력된 거래의 합계를 집계해 "예산 대비 사용액"을 보여주는 개념
- **색상 기반 시각화**로 상태 전달: 여유(초록) → 임박(노랑) → 초과(빨강)
- 매달 같은 예산 자동 반복, 필요시 그 달만 수정

### 3.4 분석 & 리포트 (대시보드 요약 + 필터 탐색)

**설계 원칙**: 대시보드가 자동으로 핵심 요약을 보여주고(수동 조작 없이), 더 보고 싶으면 카테고리·기간·태그로 필터링해 파고든다.

**대시보드 자동 요약 (들어가면 바로 보임)**
- 이번 달 총수입/총지출/잔액
- 상위 지출 카테고리 랭킹 (많이 쓴 순)
- 자주 가는 가맹점 TOP (방문 횟수·금액)
- 지난달 대비 증감 (카테고리별 +/− %)
- 튀는 지출 감지: 평소(직전 N개월 평균) 대비 비정상적으로 높은 카테고리 강조

**카테고리별 상세**
- 대분류 금액·비중 → 소분류 드릴다운 (식비 32만 → 외식 20 / 카페 8 / 식료품 4)
- 카테고리별 거래 건수·평균 단가 (예: 카페 23건, 건당 평균 3,500원)

**비교·추이**
- 이번 달 vs 지난 달 카테고리별 증감
- 카테고리별 월별 추이 그래프
- 태그 기반 집계 (데이트 23만, 정기권 6.5만 등)

**차트 시각화**
- 카테고리 비중 (도넛/파이)
- 월별 추이 (막대/선)

**필터 탐색**
- 기간 / 카테고리 / 소분류 / 태그 / 결제수단 / 가맹점 조합 필터
- 자주 보는 필터 조합을 즐겨찾기로 저장해 대시보드에 고정

**출력·내보내기**
- 월간 리포트 요약(한 페이지)
- CSV/엑셀 내보내기
- (선택) PDF 리포트

### 3.5 고정 지출 (템플릿 기반 빠른 입력)
- 반복 항목을 **템플릿**으로 등록: 이름, 금액, 카테고리, 결제수단 (월세, 구독료, 보험료 등)
- 매달 해당 템플릿을 원클릭으로 "오늘 날짜 거래로 추가" (자동 생성 아님 — 사용자가 직접 입력)
- 입력 시 금액 등 그 자리에서 수정 가능 (공과금처럼 매달 다른 항목 대응)
- 템플릿 목록 관리: 수정·삭제·활성/비활성

## 4. 데이터 입력 파이프라인 (핵심 확장 설계)

모든 입력 경로는 공통 파이프라인을 거친다. 경로별로 다른 것은 "raw 데이터를 얻는 방법"뿐이며, 정규화·검증·중복감지·저장은 공통 처리한다.

```
[수동 입력]  ┐
[CSV 파싱]   ├→ ImportSource (공통 인터페이스)
[고정지출]   │        ↓
[OCR 추출]   ┘   정규화 → 검증 → 중복감지 → Transaction 저장
[오픈뱅킹 등 향후]      ↑
                  (매핑 규칙 · 태그 부여 공통)
```

**ImportSource 인터페이스 (개념)**
- `fetch_raw()` → 경로별 원시 데이터 획득
- `normalize()` → 공통 거래 후보 형태로 변환 (날짜/금액/가맹점/메모)
- 이후 공통 검증·중복감지·저장 로직으로 위임

이 구조 덕분에 새 입력 경로를 추가할 때 인터페이스 구현만 하면 되고 기존 로직은 건드리지 않는다.

**입력 경로별 구현 우선순위**
1. 수동 입력 (기반)
2. CSV 가져오기 (카드/은행 내역, 열 매핑 기능 포함)
3. 고정 지출 자동 생성
4. (선택) 영수증 OCR

> 구조는 4가지 경로를 모두 수용하도록 처음부터 마련하고, 실제 구현만 단계적으로 진행한다.

**CSV 가져오기 세부**
- 우선 대응: KB국민은행, 우리은행 (거래내역 CSV/엑셀 다운로드 양식)
- 카드사·은행마다 다른 양식 대응: 열 매핑(어느 열이 날짜·금액·가맹점인지 지정)
- 매핑 프리셋 저장 (은행별 재사용)
- 중복 거래 감지 후 사용자 확인

## 4.5 거래 추적성 & 가맹점 자동 분류 (핵심)

가계부의 목적은 "몰랐던 지출을 눈으로 확인"하는 것. CSV의 원본 가맹점명이 "(주)○○" 식으로만 찍히면 무용지물이므로, 추적 가능하게 만드는 두 장치를 둔다.

**(a) 거래에 충분한 정보 보존**
거래마다 다음을 모두 저장한다:
- 원본 가맹점명(raw, CSV 원문 그대로)
- 별칭(정리된 이름, 예: "스타벅스")
- 카테고리 + 태그
- 메모

**(b) 가맹점명 → 카테고리 자동 매핑 규칙**
"가맹점명에 특정 키워드가 포함되면 → 해당 카테고리·별칭으로 자동 분류"하는 규칙을 저장·적용한다.

| 가맹점명 키워드 | 카테고리 | 별칭 |
|------|------|------|
| 스타벅스 | 식비 > 카페 | 스타벅스 |
| GS25 | 식비 > 식료품 | GS25 |
| 올리브영 | 쇼핑 > 뷰티 | 올리브영 |
| 무신사 | 쇼핑 > 의류 | 무신사 |

- 동작: 규칙은 처음엔 비어 있음. 처음 보는 가맹점은 사용자가 1회 분류 → "다음부터 자동 분류할까요?" 확인 → 규칙 저장
- 이후 같은 키워드가 포함된 거래는 자동 분류 + 별칭 정리
- 규칙이 쌓일수록 CSV 가져오기 시 자동 분류율 상승, "정체불명 거래" 감소
- 한계: 100% 자동 아님. 모호한 가맹점(예: 종합몰)은 사용자 판단 필요

**데이터 구조 추가: merchant_rules**
- id, user_id, keyword(매칭 키워드), category_id, alias, priority(우선순위)

## 6. 데이터 구조 (확장 대비 초안)

> 모든 주요 테이블에 user_id 포함 (단일 유저 시작, 멀티 유저 확장 대비)

**transactions**
- id, user_id, date, type(income/expense), amount, category_id, payment_method_id
- raw_merchant(CSV 원본 가맹점명), alias(정리된 이름), memo
- source(manual/csv/template/ocr), external_ref(중복감지용)

**merchant_rules** (가맹점 자동 분류)
- id, user_id, keyword, category_id, alias, priority

**categories** (계층)
- id, user_id, name, type, color, parent_id(자기참조), is_default

**tags** + **transaction_tags** (다대다)
- tags: id, user_id, name
- transaction_tags: transaction_id, tag_id

**payment_methods**
- id, user_id, name(현금/카드/계좌), type

**budgets** (추상화)
- id, user_id, period(월), limit_amount, scope_type(category/tag/total), scope_ref

**recurring_templates** (고정 지출 템플릿)
- id, user_id, name, amount, category_id, payment_method_id, is_active

**import_presets** (CSV 매핑)
- id, user_id, source_name(카드사), column_mapping(JSON)

## 7. 개발 단계 제안

1. **기반 구축**: 프로젝트 셋업(FastAPI/React/PostgreSQL), 스키마·마이그레이션, 거래 CRUD, 수동 입력
2. **분류·분석**: 카테고리 계층·태그, 차트, 검색·필터
3. **자동화**: 예산·알림, 고정 지출, CSV 가져오기(파이프라인 + 매핑)
4. **확장·편의**: OCR(선택), 내보내기/백업, 멀티 유저 인증

## 7.5 개발 · 운영 환경

| 역할 | 기기/환경 | 비고 |
|------|----------|------|
| 개발 | MacBook | 코드 작성·로컬 테스트 |
| 운영 서버 | Windows 데스크탑 | PostgreSQL + FastAPI + React 호스팅 |
| 클라이언트 | iPhone, 브라우저 | 같은 집 와이파이 내 접속 |

**핵심 원칙**
- 개발(Mac)과 운영(Windows)의 OS 차이를 흡수하기 위해 **Docker로 전 구성을 컨테이너화**. Mac에서 만든 구성을 Windows에서 동일하게 실행.
- 서버는 데스크탑이 켜져 있을 때만 접속 가능 (24시간 상시 아님). 데이터는 DB에 영구 보존되므로 꺼져도 기록은 안전.

**개발 흐름**
1. Mac에서 Docker Compose로 PostgreSQL + FastAPI + React 로컬 구동
2. 기능 구현·테스트
3. 완성분을 Windows 데스크탑에 동일 Docker 구성으로 배포

**Windows 운영 세팅**
- 실행: Docker Desktop for Windows (간단) 또는 WSL2 + Docker (리눅스 서버 환경에 근접, 학습 가치 높음)
- 부팅 시 자동 실행 설정으로 데스크탑 켜면 서버 자동 기동

**iPhone / 로컬 네트워크 접속**
- iPhone Safari에서 데스크탑 로컬 IP:포트로 접속 (예: 192.168.0.10:8000)
- 준비 사항: Windows 방화벽에서 해당 포트 개방, 데스크탑에 고정 IP 할당(공유기 DHCP 예약)
- (선택) Safari "홈 화면에 추가" + PWA 설정으로 앱처럼 사용

**향후 확장 (필요 시)**
- 외부에서 상시 접속: 미니PC/라즈베리파이 홈서버 분리, 또는 클라우드 배포(Railway/Render 등)
- 외부 접속용 터널링: Cloudflare Tunnel 등
- PostgreSQL 데이터는 덤프/복원으로 이전 용이

## 8. 확정 사항 / 남은 결정

**확정**
- 예산: 시각화(색상) 중심, 강제 알림 없음, 입력 거래 합계 집계 방식
- 고정 지출: 템플릿 등록 후 원클릭 수동 입력 (자동 생성 안 함)
- CSV 우선 대응: KB국민은행, 우리은행
- 카테고리: 9장 목록 (헬스는 문화·여가>운동·헬스, 데이트 등은 태그로 처리)
- 가맹점 자동 분류 규칙(merchant_rules) 포함 — 한 번 분류하면 이후 자동
- 거래에 원본 가맹점명·별칭 보존으로 추적성 확보
- 개발=Mac / 운영=Windows 데스크탑 홈서버 / 클라이언트=iPhone, Docker 컨테이너화
- 분석: 대시보드 자동 요약(상위 카테고리·가맹점 TOP·증감·튀는 지출) + 필터 탐색

**남은 결정**

## 9. 기본 카테고리 / 결제수단

**지출 카테고리 (대분류 > 소분류)**
- 식비 > 외식 / 식료품 / 카페·간식
- 교통 > 대중교통(정기권: 기후동행·K-패스 등) / 택시 / 주유 / 주차
- 주거·통신 > 월세·관리비 / 공과금 / 휴대폰 / 인터넷
- 생활 > 생필품 / 가전·가구 / 세탁
- 건강 > 병원 / 약국
- 문화·여가 > 영화·공연 / 여행 / 구독서비스 / 취미(드럼 등) / 운동·헬스(헬스장·보충제·운동용품)
- 쇼핑 > 의류 / 뷰티(올리브영 등) / 전자기기 / 잡화
- 경조사·기타 > 선물 / 경조사비 / 수수료

> 분류 기준: **물건(재화)을 사면 쇼핑, 경험·서비스에 쓰면 문화·여가**
> (예: 올리브영=쇼핑>뷰티, 드럼 레슨=문화·여가>취미)

**수입 카테고리**
- 급여 / 용돈 / 이자·투자수익 / 환급 / 기타수입

**결제수단**
- 현금 / 체크카드 / 신용카드 / 계좌이체 / 간편결제(카카오페이·네이버페이 등)

> 모두 기본 제공값이며 추가·삭제·수정 가능
